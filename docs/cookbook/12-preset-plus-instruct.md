# Preset + Instruct 원리

이 문서는 웹 UI의 `프리셋 기반 생성`이 내부적으로 어떻게 동작하는지 설명합니다.

특히 아래 질문에 답하기 위해 작성했습니다.

- 왜 모델이 두 개 필요한가?
- 왜 `Base`와 `CustomVoice`를 같이 고르는가?
- 저장된 프리셋 하나만으로는 왜 끝나지 않는가?
- `프리셋 그대로 생성`과 `프리셋 + 말투 지시`는 무엇이 다른가?

## 1. 먼저 결론

`프리셋 기반 생성`은 모델 두 개를 억지로 늘려 놓은 기능이 아닙니다.

이 기능은 서로 역할이 다른 두 단계를 한 번에 이어 붙인 흐름입니다.

1. `Base` 모델이 참조 음성에서 스타일 단서를 읽습니다.
2. `CustomVoice` 모델이 그 스타일 단서를 유지한 채 새 대사와 `instruct`를 적용해 말합니다.

즉, 한 모델이 모든 일을 다 하는 구조가 아니라:

- `Base`는 스타일 기준을 읽는 역할
- `CustomVoice`는 실제 발화와 말투 지시를 반영하는 역할

로 나뉘어 있습니다.

## 2. 프리셋은 무엇을 저장하는가

웹 UI에서 프리셋을 저장하면, 실제로는 아래 정보가 함께 저장됩니다.

- 참조 음성 경로
- 참조 음성 텍스트
- `clone prompt` 파일 경로
- 프리셋 이름, 언어, 메모
- 어떤 `Base` 모델로 prompt를 만들었는지

중요한 점은, 프리셋이 곧바로 “독립 실행 모델”은 아니라는 것입니다.

프리셋은 `이 스타일을 다시 쓰기 위한 기준 정보 묶음`입니다.

즉 프리셋은:

- 음색 기준
- 참조 문장
- clone prompt

를 다시 꺼내 쓰게 해주는 저장 형식이지, 모델 자체를 대체하지는 않습니다.

## 3. 왜 Base 모델이 필요한가

`Base` 모델은 참조 음성으로부터 “이 사람이 어떤 느낌으로 말하는가”를 읽는 데 쓰입니다.

이 프로젝트에서 `Base`는 주로 아래 역할을 맡습니다.

- 참조 음성 + 참조 텍스트를 사용해 `clone prompt` 생성
- 프리셋에 저장된 기준 음성을 바탕으로 스타일 기준을 다시 읽기
- 참조 음성의 음색, 속도감, 리듬, 발성 인상을 유지하기 위한 조건화

즉 `Base`가 없으면, 프리셋에 담긴 “원래 목소리의 스타일 정보”를 다시 불러오는 단계가 약해집니다.

## 4. 왜 CustomVoice 모델이 필요한가

`CustomVoice` 모델은 실제 문장을 말하고, `instruct`를 적용하는 역할을 맡습니다.

이 프로젝트에서 `CustomVoice`는 아래 역할을 담당합니다.

- 대사를 실제로 생성
- 화자를 고름
- `instruct`에 따라 감정, 분위기, 말투를 바꿈
- 스타일이 들어간 상태에서 새 문장을 읽음

즉 `CustomVoice`가 없으면:

- 프리셋의 스타일을 살린 채
- 새 대사를 만들고
- 말투 지시까지 반영하는

지금의 `프리셋 + instruct` 생성이 성립하지 않습니다.

## 5. 왜 모델 두 개가 동시에 필요한가

많이 헷갈리는 지점이 바로 여기입니다.

### 5.1 사용자가 보는 관점

사용자 입장에서는 이렇게 이해하면 됩니다.

- `Base`: 원래 목소리 스타일을 읽는 쪽
- `CustomVoice`: 그 스타일로 실제 문장을 말하는 쪽

### 5.2 내부 동작 관점

내부적으로는 다음 순서로 생각하면 됩니다.

1. 저장된 프리셋에서 기준 음성과 참조 텍스트를 읽습니다.
2. `Base`가 그 기준 음성의 스타일 단서를 다시 불러옵니다.
3. `CustomVoice`가 새 대사와 `instruct`를 함께 받아 최종 음성을 만듭니다.

즉 두 모델은 “같은 일의 중복”이 아니라 서로 다른 단계의 담당자입니다.

## 6. 그렇다면 모델 하나로 줄일 수는 없는가

경우에 따라 가능할 수도 있지만, 현재 이 프로젝트의 hybrid 경로는 그 방식으로 설계돼 있지 않습니다.

현재 구현 목표는 아래 둘을 동시에 살리는 것입니다.

- 참조 음성의 스타일 유지
- `CustomVoice`의 `instruct` 제어

이 두 가지를 동시에 잡기 위해, 현재는 `Base + CustomVoice` 조합을 사용합니다.

즉 지금 구조는 “복잡해서 두 개를 둔 것”이 아니라, 각 모델의 강점을 분리해서 쓰려는 선택입니다.

## 7. `프리셋 그대로 생성`과 `프리셋 + 말투 지시`의 차이

같은 탭 안에 두 가지 버튼이 있는 이유도 여기서 나옵니다.

### 프리셋 그대로 생성

이 경로는 저장된 스타일을 가능한 한 그대로 유지하는 쪽에 가깝습니다.

이럴 때 씁니다.

- 같은 캐릭터 목소리로 반복 생성
- 말투 지시 없이 안정적인 재사용
- 스타일 일관성을 먼저 보고 싶을 때

### 프리셋 + 말투 지시

이 경로는 저장된 스타일을 바탕으로 `instruct`까지 추가합니다.

이럴 때 씁니다.

- 같은 목소리인데 감정만 바꾸고 싶을 때
- 차갑게, 숨차게, 밝게 같은 지시를 얹고 싶을 때
- 스타일 유지와 말투 제어를 동시에 시험하고 싶을 때

## 8. 자주 생기는 오해

### 오해 1. 프리셋이 있으면 모델 선택은 필요 없어야 하지 않나?

프리셋은 스타일 정보 묶음이지, 실행 모델이 아닙니다.

그래서 실제 생성에는 여전히 어떤 `Base`, 어떤 `CustomVoice`를 쓸지 정해야 합니다.

### 오해 2. 프리셋 안에 clone prompt가 있으면 Base는 더 필요 없지 않나?

현재 hybrid 경로는 프리셋의 기준 정보를 바탕으로 `Base` 단계와 `CustomVoice` 단계를 분리해서 사용합니다.

즉 clone prompt 저장만으로 전체가 끝나는 구조는 아닙니다.

### 오해 3. 그러면 모델이 많아져서 사용자만 헷갈리는 것 아닌가?

맞습니다. 그래서 UI에서는 모델 설명보다 “왜 두 모델이 필요한지”를 먼저 보여주는 쪽이 더 중요합니다.

이 문서도 그 이유로 분리했습니다.

## 9. 어떤 조합을 먼저 쓰면 좋은가

보통은 아래 순서를 권합니다.

1. `Base 1.7B` + `CustomVoice 1.7B`
2. 프리셋 선택
3. 먼저 `프리셋 그대로 생성`
4. 그다음 같은 문장에 `instruct`를 얹어 비교

이렇게 해야:

- 프리셋 자체의 스타일
- instruct로 바뀐 결과

를 비교하기 쉽습니다.
